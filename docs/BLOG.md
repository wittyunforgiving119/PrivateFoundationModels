# I built a polyfill for Apple's FoundationModels — and then made it a runtime passthrough

**TL;DR — One Swift call site, three on-device runtimes. On iOS 26 it routes to Apple's actual native FoundationModels. On iOS 18, the same call site runs against CoreML on the Neural Engine. On a Mac dev box, it runs against any `mlx-community/*` model on the GPU. The same `respond(to:)` syntax. Verified end-to-end including the official `openai` Python SDK over an HTTP local server. ## The problem

Apple unveiled the `FoundationModels` framework at WWDC 2025. `LanguageModelSession` + `@Generable` + `Tool` is the cleanest on-device LLM API I've seen in any platform SDK. Type-safe Swift macros, structured output without praying-to-the-prompt, native tool calling, grammar-constrained sampler. It's gorgeous.

It also ships **only on iOS 26 / macOS 26 / visionOS 26**, and only routes to **Apple's 3 B on-device model** — no custom adapters beyond what Apple ships, no swapping in your own fine-tune.

If you're shipping an app today, ~80% of your iOS install base is still on iOS 18.

If you've fine-tuned your own model (Gemma 4 with your domain LoRA, a Qwen3.5 you converted to CoreML, an `mlx-community/*` repo you're testing on a dev Mac), Apple's framework won't load it.

That's two paywalls on top of a beautiful API.

## The thing I built

`PrivateFoundationModels` (PFM) is a Swift package that:

1. **Mirrors Apple's API surface verbatim.** Same type names, same initializer shapes, same method signatures. `LanguageModelSession`, `Instructions`, `GenerationOptions`, `Transcript`, `Tool`, `Generable`, `SystemLanguageModel`, `Response<T>`, `ResponseStream<T>`, `GenerationError`, `Prompt`, `@Generable`, `@Guide`, `@PromptBuilder` — all of it. Code written against Apple's framework compiles unchanged against PFM.

2. **Routes generation to a pluggable backend.** Today there are three official backend products you pick from at install time:
   - **`PrivateFoundationModelsApple`** — passthrough to Apple's actual native FoundationModels on iOS 26+.
   - **`PrivateFoundationModelsCoreML`** — runs LFM2.5, Gemma 4, Qwen3.5, Qwen3-VL, FunctionGemma, EmbeddingGemma on the Apple Neural Engine via [`john-rocky/CoreML-LLM`](https://github.com/john-rocky/CoreML-LLM).
   - **`PrivateFoundationModelsMLX`** — runs any `mlx-community/*` repo (Llama, Qwen, Gemma, Mistral, Phi, plus VLMs) on Apple GPU via [`ml-explore/mlx-swift-lm`](https://github.com/ml-explore/mlx-swift-lm).

3. **Becomes a runtime passthrough on iOS 26.** When you install both the Apple backend and the CoreML/MLX backend, you pick at runtime:

   ```swift
   if #available(iOS 26.0, *), AppleFoundationModel.isAvailable {
       SystemLanguageModel.default = SystemLanguageModel(backend: AppleFoundationModel.load())
   } else {
       SystemLanguageModel.default = SystemLanguageModel(
           backend: try await CoreMLLanguageModel.load(.lfm2_5_350M))
   }
   ```

   Everything downstream — `respond(to:)`, `@Generable`, `Tool`, `streamResponse` — is byte-identical to Apple's. Your app doesn't know which runtime it landed on.

## What was hard

I want to call out three problems that took meaningful engineering, because they're not obvious from the outside.

### Cross-protocol `@Generable`

Apple's `Generable` is generated by `@Generable` macros at compile time. PFM has its own `Generable` protocol with a `static var generationSchema: GenerationSchema` (JSON-Schema-shaped). The two are different Swift protocols.

To make `respond(to: prompt, generating: MyType.self)` work on **both** backends without forcing the user to choose, PFM's Apple backend translates PFM's `GenerationSchema` into Apple's `DynamicGenerationSchema` at runtime, feeds that to Apple's `respond(to:schema:)`, then re-serializes the resulting `GeneratedContent` back to a JSON string so PFM's existing `Generable` decoder takes over. The user's `@Generable struct` never touches Apple's framework directly.

```swift
// Sources/PrivateFoundationModelsApple/AppleFoundationModelBackend.swift
static func pfmSchemaToDynamic(
    _ schema: PrivateFoundationModels.GenerationSchema, name: String
) -> FoundationModels.DynamicGenerationSchema {
    switch schema.type {
    case "object": return .init(name: name, description: schema.description, properties: ...)
    case "array":  return .init(arrayOf: ...)
    case "string": return .init(type: String.self)
    // ...
    }
}
```

Two protocols, one runtime bridge. Same user-side code.

### Cross-protocol `Tool` with the model's tool loop running on Apple's side

Apple's `Tool` requires `Arguments: ConvertibleFromGeneratedContent`. PFM's `Tool` requires `Arguments: Generable` (PFM's flavor). Different protocols, same shape.

PFM wraps each PFM `Tool` at runtime in a `PFMToolAdapter` that conforms to `FoundationModels.Tool` with `Arguments = GeneratedContent`. The adapter's `call(arguments:)` renders the Apple `GeneratedContent` to a JSON string and dispatches through PFM's `AnyTool.invoke`, which decodes into the user's PFM `Generable` struct and calls their `func call(arguments:)`. When the user's function throws, Apple wraps the error in `ToolCallError`; PFM's backend unwraps it so the caller sees exactly the error their tool threw — bit-for-bit identical behavior to the CoreML and MLX backends.

And to keep PFM's transcript audit trail consistent, the backend snapshots Apple's `session.transcript` before and after each `respond(...)`, translates the new `.toolCalls` / `.toolOutput` entries back to PFM `Transcript.Entry` values, and ships them via a new `BackendGeneration.transcriptDelta` field. The session appends the delta before recording the final `.response`. End result: `session.transcript.entries` looks identical whether you ran through Apple FM, CoreML, or MLX.

### The runtime gap nobody told you about

I built a standardized bench (`pfm-bench-{apple,coreml,mlx}`, same prompt, same `GenerationOptions`, three timed iterations + warmup, median taken) so the cross-runtime comparison would be apples-to-apples. The numbers surprised me.

On Apple M4 Max, same Qwen3.5-0.8B repo, same prompt:

| Backend | Quant | TTFT | Throughput |
|---|---|---|---|
| MLX / GPU  | 4-bit | **43 ms** | **781 chars/sec** |
| Apple FM 3B | — | 298 ms | 253 chars/sec |
| CoreML / ANE | FP16-ish | 526 ms | 158 chars/sec |
| CoreML / ANE LFM2.5-350M | FP16-ish | 530 ms | 39 chars/sec |

**12.2× TTFT, 5× throughput.** Same model, same call site, different runtime.

Caveats up front: MLX is 4-bit quantized, the CoreML build is FP16-ish. ~3× of the throughput gap is the quant alone; the remaining ~1.7× is runtime/hardware path (GPU Metal vs ANE). The full numbers + methodology + multi-language extension are in [`docs/RUNTIME_COMPARISON.md`](https://github.com/john-rocky/PrivateFoundationModels/blob/main/docs/RUNTIME_COMPARISON.md) and [`docs/MULTILANG_BENCH.md`](https://github.com/john-rocky/PrivateFoundationModels/blob/main/docs/MULTILANG_BENCH.md).

It's the kind of result that needed to exist as a reproducible bench you can run on your own hardware, with caveats called out instead of swept under marketing copy.

## The OpenAI-compatible local HTTP server

PFM also ships an `OpenAI`-shaped local HTTP server so non-Swift codebases can drive Apple's on-device model. Started as "let me show Apple FM works from Python," ended up implementing the full chat/streaming/tools/vision/embeddings/JSON-mode surface verified against the official `openai==2.36` SDK without any client-side changes:

```python
from openai import OpenAI
client = OpenAI(base_url="http://127.0.0.1:11434/v1", api_key="not-required")

stream = client.chat.completions.create(
    model="apple-fm",
    messages=[{"role": "user", "content": "Three Swift concurrency facts."}],
    stream=True,
)
for chunk in stream:
    print(chunk.choices[0].delta.content or "", end="", flush=True)
```

Yes that does what you think it does. Apple Intelligence streamed to `curl -N` and Python's `openai` SDK, no shim, no proxy, no key.

Function calling, vision content arrays, embeddings — all verified end-to-end. Demos in [`Examples/PythonClient/`](https://github.com/john-rocky/PrivateFoundationModels/tree/main/Examples/PythonClient).

Multi-model loading (Ollama / LM Studio replacement):

```bash
pfm-serve-mlx \
  --model mlx-community/Qwen3.5-0.8B-MLX-4bit \
  --model mlx-community/FastVLM-0.5B-bf16 \
  --embedding-model sentence-transformers/all-MiniLM-L6-v2
```

The request body's `model:` field routes per-call. One process, many models, OpenAI clients pick by name.

## Verification matters

Through 27 tagged releases I obsessed over one thing: **claims have to be reproducible**. Each new capability lands with at least one of:

- A stub-backed unit test (94 passing at HEAD).
- A real-model deep matrix (`pfm-deep` / `pfm-mlx-deep` / `pfm-apple-deep` each run 14 scenarios — Generable shapes, Tool patterns, streaming, multimodal, PromptBuilder — against actual models on M4 Max).
- A captured run log in `docs/` (`pfm-vision-sample.txt`, `pfm-embeddings-sample.txt`, `pfm-serve-sample.json`, `pfm-serve-stream-sample.txt`).
- A scripted Python demo in `Examples/PythonClient/` driving the HTTP server via the actual `openai` package.

When a capability isn't end-to-end verified, the README says so. When the bench numbers come from a specific machine, the CSV row carries the hardware tag.

This is the part of the project I'm most proud of.

## What's next

- **Multi-machine bench fill-in.** Today there's one hardware tag in `docs/BENCHMARKS.csv` (M4 Max). PRs from M1 / M2 / M3 / M4 Pro / iPhone / iPad / Vision Pro would dramatically strengthen the runtime comparison. The contribution flow is one shell command — see [`CONTRIBUTING.md`](https://github.com/john-rocky/PrivateFoundationModels/blob/main/CONTRIBUTING.md). For iPhone specifically there's a one-tap SwiftUI bench app at [`Examples/PFMiPhoneBench/`](https://github.com/john-rocky/PrivateFoundationModels/tree/main/Examples/PFMiPhoneBench).

- **Grammar-constrained sampling on MLX.** The one remaining "Not a..." disclaimer in the README. `@Generable` on CoreML / MLX is currently enforced via system-prompt injection + auto-retry on parse failure. Apple FM uses Apple's native grammar sampler. Bringing the same guarantee to MLX needs token-level logit masking — that's the next big feature.

- **Qwen3-VL stateful routing on CoreML.** Currently lives in upstream `CoreML-LLM` as a separate Swift class (`Qwen3VL2BStatefulGenerator`); PFM doesn't auto-wire it yet.

- **`llama.cpp` / GGUF backend.** Opens the broader GGUF ecosystem (`unsloth/*`, `bartowski/*`) as a fourth runtime under the same call site.
